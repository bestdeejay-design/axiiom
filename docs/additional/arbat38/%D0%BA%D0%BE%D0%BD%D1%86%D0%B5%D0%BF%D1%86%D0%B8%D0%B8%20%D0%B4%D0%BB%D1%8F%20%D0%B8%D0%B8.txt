КОНЦЕПЦИЯ ПРОЕКТА
Система электронных ваучеров и ресторанный маркетплейс для аэропортов
Версия: 1.0 (для ИИ-разработчика)
Исполнитель: Аксиома
Заказчик: ГК Арбат38
================================================================================

ЦЕЛЬ ДОКУМЕНТА

Данный документ описывает концепцию и бизнес-логику системы, чтобы любой
ИИ-разработчик (или человек) мог понять полный контекст проекта, спроектировать
интерфейсы, сгенерировать прототипы и давать рекомендации по построению
клиентской, админской и менеджерской частей.

================================================================================
1. ОБЩАЯ ИДЕЯ
================================================================================

Система решает две задачи:

1. Автоматизация выдачи питания и воды пассажирам при задержках рейсов
   (электронные ваучеры вместо бумажных талонов).

2. Создание маркетплейса ресторанов в аэропорту, где пассажиры могут
   заказать еду заранее, оплатить картой или ваучером, получить бонусы.

Важно: обе задачи живут в одном приложении для пассажира и в одной
административной экосистеме. Они могут внедряться по очереди или параллельно.

================================================================================
2. РОЛИ ПОЛЬЗОВАТЕЛЕЙ
================================================================================

2.1. Пассажир (конечный пользователь)
- Не авторизован до момента получения ваучера или заказа.
- Использует мобильное приложение (iOS/Android) или PWA (для ваучеров).
- Хочет быстро получить ваучер, заказать еду, оплатить.

2.2. Сотрудник аэропорта
- Работает в веб-кабинете на ПК или планшете.
- Активирует ваучеры на конкретный рейс при задержке.
- Не управляет ресторанами, только ваучерами.

2.3. Кассир точки питания
- Работает на планшете/смартфоне с простой веб-страницей.
- Сканирует QR-код ваучера и подтверждает выдачу воды/еды.

2.4. Менеджер ресторана
- Работает в веб-админке.
- Управляет меню своего ресторана (цены, фото, наличие).
- Видит входящие заказы, меняет их статусы.

2.5. Администратор системы (заказчик)
- Работает в веб-админке с расширенными правами.
- Управляет аэропортами, сотрудниками аэропортов, ресторанами.
- Просматривает статистику и отчёты.

================================================================================
3. СЦЕНАРИИ ИСПОЛЬЗОВАНИЯ (ДЛЯ ПРОТОТИПИРОВАНИЯ)
================================================================================

3.1. Сценарий A: Пассажир получает ваучер при задержке рейса

Шаги:
1. В аэропорту объявляют: "Пассажирам рейса XXX для получения воды/еды
   перейдите по QR-коду на стойке".
2. Пассажир сканирует QR → открывается страница/приложение.
3. Вводит номер рейса и номер посадочного талона (или фамилию + код).
4. Система показывает QR-код ваучера с указанием, что выдаётся (вода или еда),
   список точек питания, срок действия.
5. Пассажир идёт в кафе, показывает QR кассиру.
6. Кассир сканирует QR через свою страницу, система показывает "Действителен",
   кассир нажимает "Погасить" и выдаёт продукт.
7. Ваучер становится недействительным.

Вариации:
- Если задержка увеличилась (с 2 до 4 часов), сотрудник аэропорта создаёт
  новый раунд ваучеров (горячее питание). Пассажир заходит снова и видит
  новый ваучер (старый уже использован или аннулирован).
- Один пассажир не может получить два ваучера одного типа на один рейс.

3.2. Сценарий B: Пассажир делает предзаказ еды в ресторане

Шаги:
1. Пассажир открывает мобильное приложение, видит список ресторанов своего
   аэропорта (определяется по геолокации или ручному выбору).
2. Выбирает ресторан, просматривает меню с ценами и фото.
3. Добавляет блюда в корзину, указывает время получения (например, через
   30 минут).
4. Нажимает "Оформить заказ".
5. Система предлагает применить активный ваучер (если есть) для скидки.
6. Выбирает оплату картой (или Apple Pay/Google Pay).
7. После оплаты получает подтверждение в приложении и push-уведомления.
8. Приходит в ресторан в указанное время, показывает QR-код заказа (или
   называет номер), получает еду.

3.3. Сценарий C: Сотрудник аэропорта активирует ваучеры

Шаги:
1. Сотрудник входит в веб-кабинет (логин/пароль, привязан к своему аэропорту).
2. Видит дашборд: форма создания ваучера и список активных рейсов.
3. Вводит номер рейса, количество пассажиров, выбирает тип задержки
   (вода или горячее питание).
4. Нажимает "Создать".
5. Система создаёт запись, генерирует общий QR-код для стойки информации
   (одинаковый для всех пассажиров этого рейса).
6. Сотрудник может распечатать этот QR или отобразить на экране.

3.4. Сценарий D: Менеджер ресторана управляет заказами

Шаги:
1. Менеджер входит в веб-админку ресторана.
2. Видит список текущих заказов с таймером, статусами (новый, готовится, готов).
3. При поступлении нового заказа – звуковой сигнал, заказ подсвечивается.
4. Менеджер нажимает "Принять", затем "Готовится", затем "Готов".
5. Когда пассажир забирает заказ, менеджер переводит в статус "Выдан".
6. Также менеджер может редактировать меню: добавлять позиции, менять цены,
   загружать фото.

3.5. Сценарий E: Администратор управляет всей системой

Шаги:
1. Администратор входит в суперадминку.
2. Может добавлять/удалять аэропорты, рестораны, сотрудников аэропортов,
   менеджеров ресторанов.
3. Просматривает дашборд с графиками: количество выданных ваучеров,
   использованных, популярные рестораны, выручка.
4. Настраивает правила: через сколько часов задержки давать воду, еду,
   лимиты сумм ваучеров.
5. Экспортирует отчёты в Excel.

================================================================================
4. ТРЕБОВАНИЯ К ИНТЕРФЕЙСАМ (ДЛЯ ДИЗАЙНА И ПРОТОТИПОВ)
================================================================================

4.1. Мобильное приложение пассажира (главный клиент)

Экраны:

- Экран входа / выбора аэропорта
  * При первом запуске – запрос геолокации.
  * Кнопка "Выбрать аэропорт вручную" (список).
  * Кнопка "Получить ваучер по задержке" (ведёт на форму рейса).

- Экран получения ваучера
  * Поля: номер рейса, номер посадочного талона (или фамилия).
  * Кнопка "Получить ваучер".
  * После успеха – отображение QR-кода крупно.
  * Текст: тип ваучера (вода/еда), срок действия, список точек питания.
  * Кнопка "Сохранить в кошелёк" (опционально).

- Главный экран (после авторизации в приложении)
  * Список ресторанов (карточки с фото, названием, рейтингом, временем работы).
  * Фильтры: по типу кухни, по терминалу, по времени работы.
  * Поиск по названию.

- Карточка ресторана
  * Фото, описание, часы работы, телефон, адрес.
  * Меню: категории, позиции с ценой и фото.
  * Кнопка "Добавить в корзину" для каждой позиции.

- Корзина
  * Список выбранных блюд с количеством и ценой.
  * Поле для выбора времени получения (скроллер времени).
  * Блок "Применить ваучер" – отображается, если у пользователя есть активные
    ваучеры в этом аэропорту.
  * Итоговая сумма со скидкой.
  * Кнопка "Оплатить" → переход на платёжный шлюз.

- Личный кабинет
  * История заказов (статус, сумма, ресторан).
  * Бонусный счёт (баллы).
  * Избранные рестораны.
  * Настройки (уведомления, смена номера телефона).

4.2. Веб-кабинет сотрудника аэропорта

- Страница входа (логин/пароль).
- Дашборд:
  * Форма "Создать ваучеры на рейс" (номер рейса, кол-во пассажиров,
    тип задержки: вода / горячее питание).
  * Список активных рейсов с возможностью закрыть.
  * Для каждого рейса – статистика: сколько ваучеров выдано/использовано.
  * Кнопка "Показать QR для стойки" (общий QR на этот рейс).

4.3. Страница кассира (планшет/телефон)

- Простая страница без лишних элементов.
- Крупная кнопка "Сканировать QR" (открывает камеру).
- После сканирования – отображается информация:
  * Тип ваучера (вода или горячее питание).
  * Номер рейса.
  * Статус: Действителен / Использован / Просрочен.
- Кнопка "Погасить" (активна только для действительных ваучеров).
- Подтверждение: "Ваучер погашен. Выдайте воду/еду".

4.4. Админка менеджера ресторана

- Страница входа (логин/пароль, привязан к конкретному ресторану).
- Вкладка "Заказы":
  * Таблица заказов с фильтрами по статусу, дате.
  * Каждый заказ можно перевести в следующий статус: новый -> принят ->
    готовится -> готов -> выдан.
  * Уведомления о новых заказах (звук + всплывающее окно).
- Вкладка "Меню":
  * Список категорий и блюд.
  * Возможность добавить/удалить блюдо, редактировать название, цену,
    описание, фото (загрузка файла).
- Вкладка "Настройки": часы работы, адрес, телефон.

4.5. Суперадминка (для заказчика)

- Управление аэропортами: список, добавление, редактирование.
- Управление сотрудниками аэропортов: привязка к аэропорту, сброс пароля.
- Управление ресторанами: добавление ресторана, назначение менеджера.
- Управление менеджерами ресторанов: создание учётки, привязка к ресторану.
- Статистика: графики по ваучерам (по дням, по аэропортам), по заказам,
  по выручке ресторанов.
- Настройки системы: пороги задержки (часы для воды, для еды), лимиты
  ваучеров в рублях (если используется денежный эквивалент).

================================================================================
5. ОСНОВНЫЕ БИЗНЕС-ПРАВИЛА (ЛОГИКА ДЛЯ РАЗРАБОТЧИКОВ)
================================================================================

5.1. Правила ваучеров
- Ваучер создаётся сотрудником аэропорта для конкретного рейса и типа (вода/еда).
- Лимит ваучеров = количество пассажиров, указанное сотрудником.
- Пассажир может получить ваучер только один раз для данного рейса и данного типа.
- Ваучер действителен в течение 24 часов с момента создания рейса или до
  фактического вылета (что наступит раньше).
- После погашения ваучер становится недействительным.
- Неиспользованные ваучеры автоматически удаляются через 24 часа.

5.2. Правила заказов в ресторанах
- Заказ может быть оплачен картой (полностью) или частично ваучером + карта.
- Ваучер применяется только к заказам в том же аэропорту, где был активирован.
- Ваучер не может быть использован для оплаты алкоголя (если это противоречит
  законодательству – уточнить с заказчиком).
- Бонусы начисляются только за оплаченную картой часть (не за ваучерную).
- Бонусы можно потратить на любой заказ, кроме оплаты ваучером.

5.3. Правила интеграции ресторанов
- Каждый ресторан может быть подключен через API кассовой системы (iiko, R-Keeper).
- Если API нет, возможен ручной режим: заказы видны в админке ресторана,
  менеджер вручную переносит их в кассу.
- Меню ресторана может обновляться через админку вручную или автоматически
  по API (если ресторан предоставляет фид).

6. ТЕХНИЧЕСКИЕ КОНТЕКСТНЫЕ ЗАМЕТКИ (ДЛЯ ИИ)
================================================================================

- Платформы: iOS, Android (нативные или Flutter/React Native), веб (React).
- Бэкенд: REST API, предпочтительно Node.js + PostgreSQL.
- Аутентификация: JWT для сотрудников, для пассажиров – сессия по номеру телефона.
- QR-коды: генерируются на сервере, содержат уникальный идентификатор ваучера.
- Платежи: интеграция с ЮKassa или Т-Банк.
- Push-уведомления: Firebase для Android, APNS для iOS.
- Геолокация: для определения аэропорта по координатам.
- Офлайн-режим: приложение должно кэшировать меню и последние ваучеры.

================================================================================
7. ПРИМЕРНЫЙ ПОРЯДОК РАЗРАБОТКИ (ДЛЯ ПОНИМАНИЯ ПРИОРИТЕТОВ)
================================================================================

1. Сначала реализуется Поток 1 (ваучеры) как минимально жизнеспособный продукт:
   - Бэкенд: создание ваучеров, получение, погашение.
   - Веб-кабинет аэропорта.
   - PWA для пассажира.
   - Страница кассира.

2. Затем параллельно или последовательно Поток 2 (ресторанный маркетплейс):
   - Мобильное приложение (iOS/Android).
   - Админка ресторанов.
   - Платёжный шлюз.
   - Бонусная система.
